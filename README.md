# Классификатор обращений для автоматизации службы поддержки

## Описание проекта

Проект представляет собой ML-сервис для автоматической классификации текстовых обращений в службу поддержки.

Сервис принимает текст обращения клиента и возвращает:

- категорию обращения;
- приоритет обработки.

Пример входных данных:

```json
{
  "text": "У меня не проходит оплата картой"
}
```

Пример ответа:

```json
{
  "category": "Оплата",
  "priority": "Высокий"
}
```

## Цель проекта

Цель проекта — разработать программный модуль, который помогает автоматизировать первую линию службы поддержки: сортирует обращения клиентов по категориям и определяет приоритет обработки.

## Категории обращений

В проекте используются следующие категории:

- Оплата
- Техническая ошибка
- Доставка
- Возврат и обмен
- Спам

## Установка и запуск

### 1. Клонировать репозиторий

```bash
git clone https://github.com/Komi-Mir/Request-classifier-for-support-service-automation.git
cd Request-classifier-for-support-service-automation
```

### 2. Установить зависимости

```bash
python -m pip install -r requirements.txt
```

### 3. Обучить модель

```bash
python model.py
```

После обучения создаются файлы:

- `model.pkl`
- `TfIdfVectorizer.pkl`
- `label_encoder.pkl`

### 4. Запустить API

```bash
python app.py
```

API запускается локально по адресу:

```txt
http://127.0.0.1:8000
```

Основной endpoint:

```txt
POST /predict
```

Пример запроса через PowerShell:

```powershell
Invoke-RestMethod `
  -Uri "http://127.0.0.1:8000/predict" `
  -Method POST `
  -ContentType "application/json; charset=utf-8" `
  -Body '{"text":"У меня не проходит оплата картой"}'
```

Пример ответа:

```txt
category priority
-------- --------
Оплата   Высокий
```

### 5. Запустить интерфейс

В отдельном терминале:

```bash
streamlit run streamlit_app.py
```

После запуска откроется веб-интерфейс, где можно ввести текст обращения и получить категорию и приоритет.

## Метрики модели

Финальная модель:

- алгоритм: TF-IDF + Logistic Regression;
- признаки: униграммы и биграммы;
- максимальное количество признаков: 5000;
- train/test split: 80/20;
- stratify по категориям: используется.

### Проверка переобучения

| Выборка | Accuracy | F1 weighted |
|---|---:|---:|
| Train | `0.9965` | `0.9965` |
| Test | `0.9236` | `0.9223` |

Метрики на train выше, чем на test, поэтому у модели есть признаки небольшого переобучения.  
При этом качество на test-выборке остаётся высоким: Accuracy ≈ `0.92`, F1 weighted ≈ `0.92`.

### Анализ ошибок

Для анализа ошибок модель дополнительно сохраняет:

- `confusion_matrix.csv` — матрица ошибок;
- `model_errors.csv` — примеры обращений, где модель ошиблась;
- `metrics_summary.csv` — сравнение train/test метрик.

Всего на test-выборке модель допустила `11` ошибок.

Примеры ошибок:

| Текст | Истинная категория | Предсказанная категория |
|---|---|---|
| всё зависнуть | Техническая ошибка | Оплата |
| обменять заказ | Возврат и обмен | Доставка |
| хотеть выиграть приз | Спам | Возврат и обмен |
| проверка система | Спам | Оплата |
| вернуть деньга заказ | Возврат и обмен | Оплата |
| abc | Спам | Техническая ошибка |

Основные ошибки связаны с короткими или неоднозначными обращениями. Например, текст `вернуть деньга заказ` может относиться и к оплате, и к возврату, поэтому модель иногда путает близкие категории.
